## Agent 使用原则

**协作与表达**

- **对话**：以中文作为对话语言，专有名词可使用英文或缩写。哪怕用户本身使用了其他语言，除非用户明确说了“用英文对话”，否则仍需尊重此规则。
- **输出**：对用户宜简洁、可执行；推理、核对与风险判断应充分，不必把全过程写给用户。
- **语气**：避免奉承式、填充式开场与收尾。

**信息与读取**

- **先读后写**：改代码或文档前，先读相关现有文件与上下文再动手。
- **避免无谓重复读取**：仍持有可靠上下文时不重复读同一文件；上下文不足或变更面扩大时应重读。
- **读取优先级**：`docs/state/` → 源代码 → `docs/changes/` → `docs/notes/`（待探索想法）→ `<spec-dir>`（仅实现阶段）。

**修改方式与范围**

- **编辑模式**：优先编辑/补丁式修改，避免无必要整文件重写；方案宜简单直接，仅改任务所需范围，避免顺手大重构。
- **旧数据兼容**：数据模型变更可能影响已有数据时（字段类型变更、删除、格式变化等），agent 不得自行决定兼容策略或编写迁移逻辑；须向用户确认是否兼容旧格式、是否需迁移脚本。默认不兼容、不迁移，除非用户明确要求。
- **脚本优先于内联命令**：需要编写脚本逻辑时（哪怕很短），应创建独立脚本文件再执行，不得将脚本内容拼接到 shell 命令中（如 `echo '...' | bash`、`bash -c '...'`、heredoc 等）。脚本文件便于阅读、调试和复用。

**Review**

- 交付文档、代码、测试等产出后，在确认完成时多问一次：「Review 我的工作，确认达到要求。」
- 用户说「review 一下」或类似表达时，默认可直接修改以解决问题，不必问「要不要改」。涉及架构、接口、数据模型等设计决策时，先与用户确认方案。可以问怎么做，不要问做不做。

## 文档与事实来源

### SSOT

- `docs/state/` + code = 系统唯一事实来源
- agent 与开发者必须优先参考 state
- spec 不构成 SSOT
- 不将旧 spec 当作当前系统事实

### 文档分层

| 层 | 目录 | 回答问题 |
|----|------|----------|
| state | `docs/state/` | 系统现在是什么（SSOT） |
| manual | `docs/manual/` | 如何理解/使用系统（面向人） |
| proposals | `docs/proposals/` | 为什么做（intent） |
| notes | `docs/notes/` | 想做什么（lightweight intent） |
| issues | `issues/` | 已确认的问题 |
| changes | `docs/changes/` | 改了什么（history） |
| specs | `<spec-dir>` | 怎么做（implementation plan） |

各层的规范详见对应目录下的 `README.md`。

实现方案（how-to-implement）通过 spec 工作流确定。无论使用哪个 Agent，都应使用 `<spec-dir>` 作为 spec 工作目录。

### `<spec-dir>` 定义

- 各 agent 平台在自己的 scope 文件中声明 `<spec-dir>` 的取值（Kiro → `kiro-scope.md`，Cursor → `cursor-scope.mdc`）
- 所有规则文件中出现的 `<spec-dir>` 以当前 agent 的 scope 文件为准
- 归档路径不变：`docs/changes/<version>/specs/`

## 项目技术栈与命令

- **完成前验证**：宣称完成或断言「已通过」前，须运行与变更相称的验证；通常至少执行全量测试，或按 spec / 任务约定运行其它检查。不得在未经核实的情况下作成功断言。
