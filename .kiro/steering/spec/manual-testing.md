# Manual Testing

手工测试的完整执行规范。

---

## 覆盖范围与定位

手工测试覆盖自动化测试难以触及的场景：CLI 交互体验、配置错误信息可读性、跨组件集成、外部系统交互。

---

## 核心原则

### 最小介入

手工测试的设计目标是将用户介入压缩到绝对最少。agent 在编排测试方案时，必须遵循以下思考流程：

1. 模拟执行：在编排脚本之前，agent 先在脑中走一遍所有 sub-task 的执行过程，识别哪些步骤需要用户介入
2. 逐项挑战：对每个需要用户介入的步骤，问自己——有没有别的办法可以避免？
3. 造工具：如果现有工具不足以自动化某个步骤，agent 应主动编写辅助脚本或工具来消除该介入点，而不是将"缺少工具"作为需要用户介入的理由
4. 补日志：如果某个步骤需要用户介入是因为无法判断执行结果，考虑在代码中补充日志输出，使脚本可以通过解析日志自动验证
5. 最终兜底：只有在确认无法通过任何技术手段自动化时，才将该步骤标记为人工介入点

### 强约束

- agent 必须实际执行手工测试步骤，不允许仅编写测试文档就标记完成
- 绝对禁止将未实际执行验证的手工测试 sub-task 标记为完成
- 如果用户明确表示跳过某个手工测试 sub-task，agent 应在 tasks.md 中保留该 sub-task 为未完成状态，并在 commit message 中注明用户选择跳过

---

## 执行流程

手工测试作为一个大的 task，按以下流程执行：

### 1. 环境准备（用户确认）

- agent 列出本次手工测试所需的全部前置条件：环境变量、外部服务、测试数据、构建命令等
- 等待用户确认环境已就绪后，才进入测试执行

### 2. Sub-task 分组与脚本编排

- 将 sub-task 按可连续自动化执行的程度分组，尽可能多的 sub-task 归入同一个测试脚本
- 分组依据：一组 sub-task 之间不需要人工介入（如手动操作外部系统、确认物理状态等）
- 每组生成一个 shell 测试脚本，脚本内部对每个 sub-task 输出 PASS / FAIL 判定并汇总结果
- 脚本本身必须全自动执行，不允许脚本内部等待用户输入
- 目标：将 N 个 sub-task 的 N 次用户确认压缩为尽可能少的交互次数

### 3. 交互节奏：脚本 → 人工介入 → 脚本

整体执行是「脚本段」和「人工介入点」交替的序列：

```text
[环境确认] → 脚本A → [人工介入] → 脚本B → [人工介入] → 脚本C → [汇总]
```

- 脚本段：agent 执行测试脚本，根据输出判定 sub-task 结果
- 人工介入点：agent 给出明确的操作指令（如移动邮件、发送 SQS 消息、触发 Webhook），等待用户确认完成后再继续下一个脚本段
- 如果所有 sub-task 都可本地执行，则只有一个脚本段，无人工介入点

### 4. 汇总

所有脚本段执行完毕后，agent 根据输出结果标记 tasks.md。

## 脚本管理

- 脚本在 agent 执行该 task 时动态生成，不需要提前编写
- tasks.md 中的 `[脚本]` / `[人工介入]` 标注仅用于指导 agent 执行时如何拆分步骤
- 脚本存放路径：`<spec-dir>/<spec-name>/tests/test-task-<N>.sh`（单脚本）或 `test-task-<N><a|b|c...>.sh`（多段）
- `<N>` 为该 task 在 `tasks.md` 中的 top-level 序号（如第 2 个 top-level task 对应 `test-task-2.sh`）
- 脚本进入版本控制，随 spec 一起归档

---

## 测试资源目录

统一存放在 `<spec-dir>/<spec-name>/tests/` 目录下（进入版本控制），随 spec 一起归档到 `docs/changes/<version>/specs/`：

- Feature 示例：`<spec-dir>/resume-deduplication/tests/`
- Release 示例：`<spec-dir>/release-0.4/tests/`
- 如果项目的主配置文件被 `.gitignore` 排除，测试用配置副本应加 `test-` 前缀以绕过忽略规则（如主配置为 `app.conf`，测试配置命名为 `test-app.conf`）；测试时复制为原始文件名
- 预置 fixture 文件（如 JSON 数据、缓存样本）
- agent 动态生成的测试脚本（`test-task-<N>.sh` 等）
- 其他手工测试所需资源
